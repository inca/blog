Introduction: Coordinate Systems

First, let‚Äôs ask ourselves: what should a ‚Äúgrid system‚Äù do?

Forget about hex for a moment. Say, you want a square grid system for your game (you know, the good ol‚Äô orthogonal Cartesian system that all our games use already). Would you go around wondering how to create a ‚Äúsquare grid system‚Äù?

Probably not, because you sort of have everything you need already, specifically:

  • you can place the objects on your ‚Äúgrid‚Äù; for example, if your grid is on the XZ ‚Äúfloor‚Äù, then you can just set the Y to zero;
  • you know how to obtain the coordinates of the objects on your grid, e.g. by plucking some components of Vector3 into Vector2;
  • if you want objects to snap to grid cells: at runtime all you need to do is to Mathf.Round the coordinates, whilst in the editor you can simply drag with Control/Command key pressed;
  • finally, you may want to actually draw the grid lines on your board/floor ‚Äî¬†these are usually drawn with either a tileable texture applied on something like a primitive plane, or a shader (or any combination of those).

So there‚Äôs little point in actually implementing a ‚Äúsquare grid system‚Äù, chiefly because you already have Vector2 that makes all the above possible.

Why is hex a different story, then?

When presented with a hex grid, we (humans) tend to immediately recognize rows and columns. We‚Äôre so used to orthogonal systems in our lives that it‚Äôs natural to apply the paradigm we‚Äôre so familiar with to all the cases.

Let‚Äôs try and put our orthogonal axes onto a hex grid.

Now you can see two obvious problems with this approach:

  • the width of a regular hexagon doesn‚Äôt match its height; depending on which of the two dimensions is chosen as ‚Äúunit‚Äù, the other one will be irrational to this unit;

  • the Y axis doesn‚Äôt reach the ‚Äúmiddle dot‚Äù of every other row (one other way to look at this problem is: every other row have horizontal offset)

Trying to derive a decent logic to address hex cells with orthogonal coordinates is just plain painful. It‚Äôs time to admit that orthogonal system is just not fit for purpose ‚Äî¬†and to look for some other, more suitable system.

Luckily, there is just one that works perfectly with hexagons. Behold, the oblique cartesian system (also known as ‚Äúaxial‚Äù or ‚Äúskewed‚Äù):

Yes, it‚Äôs pretty much like the orthogonal one, except that the axes are at 60Àö to each other.

The names of the axes Q and R are chosen to conform to Amit‚Äôs hex guide.

And this makes the whole difference. Notice how both axes go through the middle dots of each ‚Äúcolumn‚Äù and ‚Äúrow‚Äù. Essentially, linear combinations of %%\vec q%% and %%\vec r%% unit vectors cover the entire hex space.

So implementing a ‚Äúhex grid system‚Äù boils down to creating a Vector2 equivalent for oblique coordinates (I‚Äôll refer to them as ‚Äúhex coordinates‚Äù) with conversions to and from our standard orthogonal coordinates.

Everything else would be implemented on top of it. For example, snapping objects to hex grid would be just a matter of getting the rounded hex coordinates and coverting them back to the World coordinates.

Implementation

Before jumping to the code, we have some choices to make.

  1. Orientation: should each axis go through the vertex of a hexagon or through its side?

  2. Scale: how much space should a single cell occupy?

So. Hexagons have two radiuses:

  • circumradius %%R%% ‚Äî the distance from center to vertex, which is also equal to the side of hexagon,
  • inradius %%r%% ‚Äî the distance from center to side, which is %%{\sqrt 3 \over 2} R%%.

In the GIF above:

  • orange hexagon has circumradius %%R = 1%%
  • blue hexagon has circumradius %%R = 0.5%%
  • green hexagon has inradius %%r = 0.5%%

I went for the last option (the one with inradius %%r = 0.5%%), because this way a primitive unit sphere fits perfectly into a single hex grid and nothing overlaps.

This also means that both axes go through the side of the hexagons, and not though their vertices.

As for the orientation, I went with aligning Q axis with World X. This one is quite arbitrary, but if it helps: Blender creates cylinders (which you can turn into perfect hexagonal prisms) in this same orientation by default, so you don‚Äôt have to rotate anything.

Put Together

With all the above in mind let‚Äôs create our Hex class stores axial coordinates and converts them into world coordinates.

// Hex.cs
using System.Collections.Generic;
using UnityEngine;

[System.Serializable]
public struct Hex {

    public static float RADIUS = 0.5f;
    public static Vector2 Q_BASIS = new Vector2(2f, 0);
    public static Vector2 R_BASIS = new Vector2(1f, Mathf.Sqrt(3));
    public static Vector2 Q_INV = new Vector2(1f / 2, - Mathf.Sqrt(3) / 6);
    public static Vector2 R_INV = new Vector2(0, Mathf.Sqrt(3) / 3);

    public static Hex FromPlanar(Vector2 planar) {
        float q = Vector2.Dot(planar, Q_INV) / RADIUS;
        float r = Vector2.Dot(planar, R_INV) / RADIUS;
        return new Hex(q, r);
    }

    public static Hex FromWorld(Vector3 world) {
        return FromPlanar(new Vector2(world.x, world.z));
    }

    public int q;
    public int r;

    public Hex(float q, float r) :
        this(Mathf.RoundToInt(q), Mathf.RoundToInt(r)) {}

    public Hex(int q, int r) {
        this.q = q;
        this.r = r;
    }

    public Vector2 ToPlanar() {
        return (Q_BASIS * q + R_BASIS * r) * RADIUS;
    }

    public Vector3 ToWorld(float y = 0f) {
        Vector2 planar = ToPlanar();
        return new Vector3(planar.x, y, planar.y);
    }

    public override bool Equals(System.Object obj) {
        Hex hex = (Hex)obj;
        return (q == hex.q) && (r == hex.r);
    }

    public override int GetHashCode() {
        return q * 37 + r * 31;
    }

    public override string ToString() {
        return "(" + q + ";" + r + ")";
    }

}

With Equals and GetHashCode implemented you can use your hexes as Dictionary keys, or put them into HashSets ‚Äî¬†very handy for things like breadth-first search. (Also, sorry if it‚Äôs obvious)

Here we have static methods Hex.FromPlanar and Hex.FromWorld ‚Äî these allow obtainining Hex from World coordinates (Vector2 and Vector3 respectively). Instance methods hex.ToPlanar and hex.ToWorld do the reverse and convert hex coordinates to World coordinates.

I won‚Äôt go into any detail with regards to the actual conversion ‚Äî¬†this stuff is brilliantly explained in the Amit‚Äôs hex guide. Just note how the orientation and scaling decisions are encapsulated in a bunch of static variables at the top (Q_BASIS, R_BASIS, Q_INV, R_INV) ‚Äî¬†so if you need to change the orientation, these are the things you‚Äôd need to tweak.

Snapping Objects

With the essentials in place we can now implement a component for snapping objects to hex grid roughly as follows:

  1. convert transform.position to hex using Hex.FromWorld(transform.position);
  2. convert hex back to Vector3 using hex.ToWorld().

Hex coordinates are integers, so rounding will provide us with that snap we were looking for.

Let‚Äôs call our component HexSnap.

// HexSnap.cs
using System.Collections;
using System.Collections.Generic;
using System.Linq;
using UnityEngine;

[ExecuteInEditMode]
public class HexSnap : MonoBehaviour {

    public Hex hex {
        get {
            return Hex.FromWorld(transform.position);
        }
    }

    public Hex localHex {
        get {
            return Hex.FromWorld(transform.localPosition);
        }
    }

    public void ApplyTransform() {
        Vector3 newPos = this.localHex.ToWorld(0f);
        transform.localPosition = newPos;
    }

    #if UNITY_EDITOR
    void Update() {
        if (!Application.isPlaying) {
            ApplyTransform();
        }
    }

    void OnDrawGizmosSelected() {
        UnityEditor.Handles.Label(transform.position, hex.ToString());
    }
    #endif

}

Now attach it to your game object and observe how it snaps onto hex grid floor on XZ plane.

Neighbours

Each hex cell has 6 neighbouring cells.

Let‚Äôs list the neighbours of ‚Äúcell zero‚Äù starting with the rightmost one, in counter clockwise direction.

// Hex.cs
// ...
    public static Hex[] AXIAL_DIRECTIONS = new Hex[] {
        new Hex(1, 0),
        new Hex(0, 1),
        new Hex(-1, 1),
        new Hex(-1, 0),
        new Hex(0, -1),
        new Hex(1, -1),
    };

So to find all neighbours of any given hex all we need to do is to loop through these axial directions and add corresponding q and r components.

The only thing missing is component-wise arithmetics of Hex instances, so let‚Äôs add them.

// Hex.cs
// ...
    public static Hex zero = new Hex(0, 0);

    public static Hex operator +(Hex a, Hex b) {
        return new Hex(a.q + b.q, a.r + b.r);
    }

    public static Hex operator -(Hex a, Hex b) {
        return new Hex(a.q - b.q, a.r - b.r);
    }

Awesome! Now let‚Äôs add few methods to our Hex:

  • Hex GetNeighbour(int dir) to get the neighbouring cell in specified direction,
  • IEnumerable<Hex> Neighbours() to get all cell neighbours.
// Hex.cs
// ...
    public Hex GetNeighbour(int dir) {
        Hex incr = AXIAL_DIRECTIONS[dir % AXIAL_DIRECTIONS.Length];
        return this + incr;
    }

    public IEnumerable<Hex> Neighbours() {
        foreach (Hex dir in AXIAL_DIRECTIONS) {
            yield return this + dir;
        }
    }

Finally, let‚Äôs visualize the neighbours. Let‚Äôs create a HexNeighbours which would draw gizmos on all neighbouring hex cells:

// HexNeighbours.cs
using UnityEngine;

[ExecuteInEditMode]
public class HexNeighbours : MonoBehaviour {

    public Hex hex {
        get {
            return Hex.FromWorld(transform.position);
        }
    }

    void OnDrawGizmosSelected() {
        Gizmos.color = Color.yellow;

            Gizmos.DrawSphere(neighbour.ToWorld(), .25f);
        }
    }

}

Other Algorithms

If you‚Äôre still with me, then by now you should be getting the feel for it.

Once again I‚Äôd like to point you to the most awesome hex grid guides ever created ‚Äî from here you can choose an algorithm that is relevant to the game you‚Äôre making and port it to our hex grid system without much problem.

As an exercise, let‚Äôs implement the hex rings traversal.

// Hex.cs
// ...

    public static IEnumerable<Hex> Ring(Hex center, int radius) {
        Hex current = center + new Hex(0, -radius);
        foreach (Hex dir in AXIAL_DIRECTIONS) {
            for (int i = 0; i < radius; i++) {
                yield return current;
                current = current + dir;
            }
        }
    }

    public static IEnumerable<Hex> Spiral(Hex center, int minRadius, int maxRadius) {
        if (minRadius == 0) {
            yield return center;
            minRadius += 1;
        }
        for (int r = minRadius; r <= maxRadius; r++) {
            var ring = Ring(center, r);
            foreach (Hex hex in ring) {
                yield return hex;
            }
        }
    }

As before, let‚Äôs visualize it by creating a dummy component and drawing some gizmos.
